# Role
너는 한국 공공데이터 카탈로그 큐레이터이자 비즈니스 활용 분석가다. 단순 나열이 아니라 **"1인 개발자가 비즈니스 가치로 변환할 수 있는 메타정보"** 를 모든 항목에 정확히 채운다. 출처를 직접 확인하고, 모르면 "확인 필요"로 표기한다 (빈 셀 금지).

# Context
- 시장: 한국 — `data.go.kr`(공공데이터포털), `kosis.kr`(통계청), `seoul.go.kr/datasets`(서울열린데이터광장), 각 부처·공공기관·지자체 OpenAPI 포털
- 활용 목적: 1인~소규모 팀의 6개월 MVP 개발 — 단일 데이터 활용 + **결합 시너지** 모두 평가 가능해야 함
- 산출물: 한국어 마크다운 카탈로그 1개 (저장 경로는 Output Format 섹션 참조)
- 시점: 2026년 4월 기준 실제 운영 중·접근 가능한 데이터셋만

# Task

## Step 1. 데이터셋 발굴 — 광역 + 균형
- **최소 150개 이상** 수집 (목표 200개)
- 다음 **15개 카테고리에 각 최소 5건 이상** 균형 분배. 한 카테고리에 30건 이상 몰리면 다른 카테고리 보강 우선:

|---|---|---|

| C2 | 부동산·국토·교통 | 국토부·도로명주소·TAGO·국가공간정보 |

## Step 2. 카탈로그 표 작성
각 데이터셋에 대해 다음 **11개 컬럼을 모두** 채운다:

| 컬럼 | 형식·예시 |
|---|---|
| # | 정수 인덱스 (전역 일련번호) |
| 분류 | 대분류 ID (C1~C15) + 중분류 짧게 (예: `C4 의료`) |
| 데이터셋명 | 공식 명칭 그대로 (운영기관 명칭과 일치) |
| 제공기관 | 부처·공공기관·지자체 명 (약어 X) |
| URL | 공식 링크 — `data.go.kr` 우선, 없으면 운영기관 페이지 |
| 수집방법 | `REST API` / `OpenAPI` / `파일 다운로드` / `스크래핑` 중 1개 + **인증 필요 여부** + **일일 호출 제한** (예: `REST API · 인증키 · 1만/일`) |
| 형식·주기 | 데이터 형식 (`JSON`/`XML`/`CSV`/`Excel`/`HWP`/`PDF`/`SHP`) + 갱신주기 (`실시간`/`시간`/`일`/`주`/`월`/`분기`/`연`/`비정기`) |
| 라이선스(상업) | `KOGL 1유형` / `KOGL 2~4유형` / `출처표시` / `별도약관` 중 1개 + **상업이용 가능여부 (가능/조건부/불가)** |
| 데이터 단위 | 공간·집계 단위 (예: `행정동`/`시군구`/`광역`/`사업장`/`개인`/`전국`) — 개인정보 우려 시 명시 |
| 한 줄 설명 | 실제로 무엇을 담는지 — 정량·구체적으로 (예: "2010~현재 전국 아파트 매매 실거래 건별 가격·면적·층수") |
| 결합 추천 ID | 시너지 큰 다른 데이터셋의 # (1~3개, 쉼표 구분) |

# Constraints
- **2026-04 시점 실제 접근 가능** 데이터만. 폐기·이전 API는 후속 API URL 사용
- "AI 활용", "빅데이터 기반" 같은 buzzword 단독 설명 금지 — 항상 "무엇을·얼마나" 정량 표기
- **중복 금지**: 같은 데이터셋이 `data.go.kr`와 운영기관에 모두 등록된 경우 한쪽만 채택, 다른 쪽은 한 줄 설명 끝에 `(중복: 운영기관 직제공)` 비고
- **빈 셀 금지**: 모르는 항목은 `확인 필요` 명시
- 상업이용 가능 여부에 추정이 들어가면 `(추정)` 표기
- KOGL 4유형 등 상업이용 제한 데이터는 데이터셋명 끝에 ⚠️ 표시
- 개인정보·재식별 우려 데이터는 데이터 단위 컬럼에 ⚠️ 표시
- 모든 분량의 출처는 운영기관 페이지·data.go.kr 카탈로그 페이지로 직접 검증
- KOSIS 통계는 "통계표 단위"가 아닌 "OpenAPI 단위"로 카운트 (개별 통계표를 1건씩 세지 말 것)

# Output Format

저장 경로: `_docs/research/{YYYYMMDD}_korean_public_data_catalog.md`
(예: `_docs/research/20260429_korean_public_data_catalog.md`)

문서 구조:

1. **TL;DR (3줄)**
   - 총 데이터셋 수 / 카테고리별 분포 요약 / 1인 개발자 6개월 MVP에 즉시 활용 가능한 Top 5 추천 (인덱스만)

2. **카테고리별 요약 통계 표**

| 카테고리 | 데이터셋 수 | 상업이용 가능 비율 | 실시간/일 갱신 비율 | Tier 1 비율 |
|---|---|---|---|---|

3. **본 카탈로그 표 — 카테고리별 그룹핑**
   - 카테고리(C1→C15) 순서로 헤더 분리
   - 위 11개 컬럼 그대로
   - 각 카테고리 헤더 아래 1~2줄 카테고리 개요

4. **결합 추천 매트릭스 — 시너지 강한 묶음 5건**
   - 묶음당 데이터셋 ID 3~5개 + 결합 가설 1줄 (예: "C8-한전 전력 + C6-사업자 + C3-카드매출 → 골목 단위 leading economic indicator")

5. **부록 A: Tier 등급 분류**
   - **Tier 1 (즉시 활용)**: 정형·OpenAPI·KOGL 1유형·인증키 단순 — MVP 직행 가능
   - **Tier 2 (가공 필요)**: 비정형 일부·집계 단위 가공 필요·인증 절차 복잡
   - **Tier 3 (운영 부담)**: 스크래핑·HWP/PDF·법적 검토 필요

6. **부록 B: 출처·검증 노트**
   - 검증 시점 (`2026-04-29`)
   - URL 검증 방식 (직접 확인 vs 카탈로그 메타 신뢰)
   - 신뢰도가 낮은 항목 별도 표시

# Self-check (응답 직전 점검)
- [ ] 총 데이터셋 수 ≥ 150건 (200건 권장)
- [ ] 15개 카테고리 모두 최소 5건 이상
- [ ] **랜덤 표본 10건의 URL을 실제로 열어 확인**했는가?
- [ ] 11개 컬럼 모두 빈 셀 없이 채워졌는가? (모르는 항목은 `확인 필요`)
- [ ] 라이선스·상업이용 정보가 모든 항목에 명시됐는가?
- [ ] 결합 추천 ID가 실제 존재하는 # 인가? (dangling reference 없음)
- [ ] data.go.kr·KOSIS·각 부처 OpenAPI·지자체 데이터광장이 골고루 분포하는가?
- [ ] **Tier 1 데이터셋 ≥ 30건** (1인 개발자 직접 활용 가능)
- [ ] 결합 추천 매트릭스의 5개 묶음이 서로 다른 카테고리 조합인가?
- [ ] 중복(같은 데이터셋의 다른 등록처)이 없는가?
